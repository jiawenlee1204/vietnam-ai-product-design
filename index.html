01

项目背景与目标

AI 答题工具已经非常拥挤。Gauth 日活 330 万，Photomath 下载过亿，ChatGPT 本身就能解大部分中学数学题。在这个市场里再做一个"拍照→解题"的产品，不构成差异化。

但有一件事现有工具都没做好：针对特定国家、特定考试、特定题型，输出符合当地教学规范的解答。越南 2025 年启动了新高考结构，题型变了，学生和老师都在适应期，这是一个具体的产品机会窗口。

本项目要验证的核心命题：在"越南高考数学"这个足够窄的场景里，一个编码了本地题型规则和教师评分标准的产品，能不能比通用 AI 工具输出更稳定、更被教师认可的答案。如果能，就继续做；如果不能，趁早换方向。

02

市场机会：为什么是越南高考数学

2.1 越南高考：一个足够大且正在变化的场景

越南每年约有 100 万考生参加高中毕业考试（Kỳ thi tốt nghiệp THPT），数学是必考科目。2025 年起，越南教育部按 2018 普通教育课程（GDPT 2018）改革了考试结构——这不是微调，而是题型层面的结构性变化。

考试结构变化
  • 数学卷不再只有传统单选题
  • 新增"判断式客观题"（对/错判断，每小题多组判断）
  • 新增"短答式客观题"（填写数值，有精度要求）
  • 三种题型对答题策略和时间分配的要求完全不同
  • 教育部已对约 5000 名学生试测新结构并公布参考题
这意味着什么
  • 学生和教师都在适应新题型，市面上针对新结构的备考资源有限
  • 传统补习机构和辅导书尚未完全更新
  • 通用 AI 工具不区分题型，更不了解新结构下的答题节奏
  • 这是一个有时间窗口的产品机会——适应期不会持续太久

2.2 市场规模估算（自下而上）

大市场数据（全球 K12 STEM $963 亿、东南亚在线教育 $20.6 亿）用来说明趋势，但对判断这个产品是否值得做没有太大意义。真正有用的是自下而上的估算：

层级数据来源 / 推算
越南年高考考生约 100 万人越南教育部历年公布数据
12 年级在校生约 100-110 万含未参考学生
使用过拍照搜题工具的比例估计 30-50%参考 Gauth 越南累计约 370 万次下载、智能手机渗透率 >80%
潜在可触达用户30-50 万 / 年12 年级 × 用过拍照搜题

若 5% 转化付费 × $3/月约 $4.5-7.5 万/月纯估算，需验证

坦率说，这个市场的天花板不算高。但对于一人项目，不需要一个百亿市场——需要的是一个足够具体、竞品做不好、自己能做出差异的场景。如果 MVP 跑通，扩展到 10-11 年级、其他理科、或者印尼市场，才是真正的增长空间。

2.3 为什么先做数学而不是理化

维度数学物理 / 化学
标准答案明确，便于自动评估涉及单位、实验情境、多步推理，模糊地带多
题型标签可按函数、概率、解析几何、导数等知识点结构化需处理图像、实验装置图、化学式，标签成本更高
错误定位计算错误、公式错误容易定位错误可能来自题意理解、物理概念、化学方程式配平
用户感知"这道题我算对了没有"——反馈非常直接理解型反馈更复杂，用户不容易判断 AI 是否正确

2.4 Gauth 已经在越南了——窗口期有多久？

必须正视一个事实：Gauth 在越南累计下载约 370 万次（占其全球 9.48%），2026 年 4 月已在越南上线 Live Tutor 实时导师功能。字节跳动有资源、有流量（TikTok 粉丝 82 万）、有迭代速度（年均更新 29 次）。

窗口期判断

乐观估计 6-12 个月。Gauth 的策略是"广覆盖"——163 个地区、30+ 学科、50+ 语言，它不太可能为越南高考数学一个科目做深度定制。但如果越南市场增长明显，它完全有能力快速调拨资源。

这意味着两件事：① 我们不能慢慢做，8 周 MVP 的时间线已经不算宽裕；② 我们的差异化不能只停留在"prompt 写得好"这个层面，需要在窗口期内积累 Gauth 不容易快速复制的东西——题型标签体系、教师评测数据、真实学生的错题模式。

03

用户问题与核心场景

目标用户是正在准备高考数学的越南 12 年级学生。他们的核心需求不是"系统讲得越完整越好"，而是刷题时能快速判断对错、看懂关键步骤、搞清楚自己错在哪里。

3.1 用户遇到的真实问题

看不懂解析
  • 教辅解析经常跳步
  • 不知道第一步为什么这么做
  • 缺少同类题之间的联系
拿到答案但没拿到方法
  • 拍照搜题给了结果
  • 但没说清楚这道题的"题型入口"是什么
  • 换一道类似题还是不会
新题型不适应
  • 判断式和短答式的处理方式和传统单选不同
  • 时间分配策略变了
  • 需要区分"快速验算"和"仔细推导"
不确定 AI 对不对
  • AI 给的答案看起来合理但不一定对
  • 步骤可能不符合越南教师的教法
  • 出错时没有纠正机制

3.2 首期只覆盖三个场景

场景用户在做什么产品该做什么怎样算做好了
刷题验算拍照或输入一道数学题识别题干和题型，给出答案和核心步骤学生 30 秒内能判断自己做对没有
看懂思路点击"看完整解析"按越南教学习惯解释关键步骤和知识点学生能说出这道题的解题入口
错题复盘标记"我做错了"或"没看懂"生成错因分析、同类题入口、一道变式练习学生愿意把这个当错题本来用

产品定位与目标用户

一句话说清楚这个产品是什么

越南高考数学的 AI 刷题搭档。帮你在练习时快速判断对错、看懂每道题的解题入口、把做错的题变成下次不会再错的经验。

5.1 用户角色

谁想要什么担心什么产品怎么回应
12 年级学生刷题时快速验算、看懂关键步骤、考前提效AI 会不会算错，步骤是不是太绕置信提示、简洁步骤、错题归类
数学教师学生能理解规范方法，减少重复讲解AI 是不是在帮学生抄作业教师抽样评测、学习模式优先规范解法
家长孩子学习效率提高、减少无效补习开销是否值得付费，是否只是偷懒工具学习记录、错题进展可见

5.2 产品边界

这个产品不是"替代老师"，也不是"无限制给答案的作弊器"。定位是考试练习助手：先帮学生完成自查，再帮学生复盘错因。"自查→复盘"这个定位既能降低作弊感，也更容易向教师和家长解释价值。

6.2 MVP 功能清单

MVP 拍照 / 文字输入 + 题干确认

拍照是获取题目最低摩擦的方式，但实际使用中经常会因为光线、角度、手写字迹导致识别错误。所以 MVP 必须加一步：识别完题干后让学生确认或编辑。这一步看似多余，实际上能大幅减少"因为题都读错了所以答案也错了"的情况。

MVP 题型自动识别

在给出答案之前，系统先判断这道题属于哪种题型（单选、判断式、短答式），再决定输出策略。单选题强调快速排除和选项验证；判断式题目需要逐项逐条验证；短答式题目需要精确到数值和单位。这是和通用 AI 工具最直接的差异点。

MVP 双视图输出：快速验算 / 完整解析

同一道题，两种看法。快速验算：先给答案和核心步骤，适合刷题时快速对答案。完整解析：从题意分析开始，讲知识点、推导过程和易错点，适合不会做时学习思路。注意：不叫"考试捷径"，避免给人"帮你作弊"的印象。

MVP 用户纠错入口（运营机制，非功能卖点）

每道题底部放三个按钮："答案有误""步骤看不懂""题目识别错了"。这不是一个面向用户宣传的功能，而是产品质量迭代的基础设施。后台按题型、知识点、模型版本记录所有反馈，这些数据是优化 prompt 和发现系统性问题的依据。

说清楚一件事："用户纠错入口"和"AI 帮用户诊断错因"是两件完全不同的事。前者是用户告诉我们 AI 哪里不对（bug report），后者是 AI 告诉用户你哪里做错了（错题复盘功能）。前者在 MVP 做，后者放在 V1。

V1 错题本与同类题推荐

学生标记错题后，系统自动生成错因标签（例如"导数求解步骤遗漏""概率分布公式混淆"）、复习建议、以及 1 道同类变式练习。这个功能才是长期留存和付费转化的关键——用户不会为单次答题付费，但可能为一个持续积累的错题本付费。

07

差异化设计

本项目不能把“本地化 prompt”当作壁垒。大厂可以快速雇佣本地教师、购买题库、扩充提示词。可持续差异化应来自一套持续积累的质量系统：题型标签、标准答案、教师评分、用户错因和模型迭代记录。

壁垒分层

容易被复制
  • 多语言输出
  • 拍照解题
  • 一次性教师访谈的结论
  • 写几套 system prompt
  • 声称"我们更懂本地"
需要时间积累、不容易快速复制
  • 越南高考数学题型标签体系
  • 100+ 道经教师标注的评测集
  • 按题型统计的模型错误率和修正记录
  • 真实学生的错因模式和复习行为数据
  • 可对外展示的教师认可度报告

冷启动困境的正面回应

这里有一个绕不开的鸡生蛋问题：上面说的"真实壁垒"（错题数据、教师评测、题型标签）都需要用户量来积累，但用户量需要产品先足够好——而产品刚上线时唯一能依赖的就是 prompt 质量，偏偏 prompt 又不是壁垒。

怎么破

诚实的答案是：MVP 阶段的差异化确实脆弱，这没办法假装不是。策略是在窗口期内尽快走完"prompt 质量 → 教师认可 → 种子用户 → 行为数据 → 壁垒"这条链。具体来说：

第 1-2 周：prompt 是唯一武器，尽可能把"越南高考数学"这一个场景做到比 Gauth 明显好（教师盲评高出 10%+）。这段时间差异化是脆弱的，但也没关系——Gauth 不会因为你上线就立刻抄你。

第 3-5 周：跑通质量闭环（教师评分 → 题型错误率 → prompt 修正），开始积累 Gauth 没有的数据。

第 6-8 周：有了 30-50 个真实学生的使用数据后，开始积累错题模式和复习行为——这是 Gauth 从外部无法获取的。

如果 8 周后这条链跑不通（教师不认可、学生不留存），说明这个方向本身有问题，应该及时止损。

壁垒分层

层级强度怎么建风险
Prompt 规则低根据题型和输出模板编写提示词容易被复制，且随模型更新失效
题型标签体系中把高考数学题拆为题型 × 知识点 × 解法入口 × 常见错因需要持续维护和更新
教师标注评测集中高建立标准答案、推荐解法、可接受表达和评分规则样本太小会有偏差
学生错题行为数据高记录真实学生的错因分布和复习路径需要足够用户量 + 合规数据处理
品牌信任高但慢通过教师背书、准确率报告、考试季口碑一旦答案频繁出错，信任很难恢复
什么情况下应该放弃"本地化 AI 答题"这个方向

如果教师盲评显示我们的输出和 Gauth 没有显著差异（< 10%），或者学生并不在意"题型识别"和"本地化步骤"（D7 留存 < 15%），说明这个差异化假设不成立。那时候更适合转向轻工具，比如"高考数学错题整理器"或"教师出题助手"。

9.3 Coding Agent 怎么用、各自能做什么

作为一人开发，选对工具很关键。以下是三个工具各自的能力和局限——不是"什么都行"，而是"在什么场景下最好用"。

🧠 Claude.ai（对话界面）— 用在动手写代码之前

什么时候用：项目最前期（Week 1-2），以及任何需要"想清楚"的时刻。

具体做什么：写越南高考数学的 system prompt，然后直接在对话里粘贴真题测试——看 AI 返回的答案是否正确、步骤是否符合越南教学规范、题型识别是否准确。反复迭代 prompt 直到满意。也用来做产品方案的推敲、文档撰写、竞品分析等需要大量思考的工作。

✅ 擅长：所有"策略型"工作。Prompt 调试、方案推演、文档生成、规则编写。你可以把它理解为一个随时在线的高级产品顾问 + prompt 工程师。
❌ 做不到：不能操作你本地的项目文件。不能同时看到你项目的多个文件然后做跨文件修改。不能运行代码或预览界面效果。

⌨️ Cursor — 日常开发的主力工具

什么时候用：核心开发阶段（Week 3-6），贯穿始终。

具体做什么：在 Cursor 里打开你的 Next.js 项目，用自然语言描述你想要什么（"写一个拍照上传组件，支持调用手机摄像头"），它会直接在你的项目文件里写代码。它能看到你整个项目的目录结构和文件内容，所以改一个组件时它知道哪些地方也需要同步改。前端页面、API 路由、Tailwind 样式、Supabase 接入——80% 的日常开发都在这里完成。

✅ 擅长：理解整个项目上下文后写代码、改代码、debug 报错。你改了一个数据结构，它能自动找到所有引用的地方一起改。实时预览，改完就能看到效果。
❌ 局限：不擅长"从零规划"——先在 Claude.ai 里想清楚架构再来 Cursor 写，效果会好很多。复杂的多步骤任务建议拆成小步骤一个一个做，一次让它做太多事容易出错。每次改完建议自己看一遍再 commit。

🖥️ Claude Code（命令行工具）— 搭骨架和做重活

什么时候用：项目初始化（Week 3 开头），以及需要跨文件批量操作时。

具体做什么：比如一句话让它"创建一个 Next.js 项目，配好 Tailwind + KaTeX + Supabase + Claude API，搭好目录结构和基础路由"，它会自己创建文件、安装依赖、写配置代码。也适合做评测脚本（"读取 test_set.json 里的题目，逐条调用 API，把结果存到 results.csv"）、批量重构（"把所有组件里的 useState 换成 zustand"）。

✅ 擅长：自主完成需要多步骤协调的任务。能直接操作文件系统和命令行，适合"从无到有建项目"或"一次性改很多文件"的场景。
❌ 局限：没有界面预览——写完代码需要你自己打开浏览器看效果。不如 Cursor 适合做 UI 微调。命令行操作有可能改到不该动的文件，一定要配合 git 做版本控制。
推荐节奏

① Week 1-2：Claude.ai 写 prompt、测题、定方案

② Week 3 第一天：Claude Code 一次性搭好项目骨架

③ Week 3-6：Cursor 日常写页面、接 API、调样式、修 bug

④ 穿插使用：需要批量操作或写脚本时切回 Claude Code，需要重新审视 prompt 或方案时切回 Claude.ai

10.1 主要风险

风险严重程度对策什么情况下该停
本地化优势不明显高教师盲评对比 Gauth 和 ChatGPT，按题型定位具体差距教师评分优势连续两轮 < 10%

10.2 八周路线图

1
准备期：规则梳理 + 教师关系建立
Week 1-2

Week 1 和 Week 2 是并行的两条线。一条线是你自己能做的：整理越南高考数学的三种题型结构、收集 100-150 道公开样题和历年真题、在 Claude.ai 里写 prompt 并用真题测试。另一条线依赖外部：联系 2-3 位越南数学教师（优先从之前的调研关系中找），请他们确认题型分类、提供标准答案参考、约定后续抽样评分的合作方式。

教师关系的建立需要时间，所以 Week 1 就要开始联络，不能等到 prompt 写好了才去找人。如果两周内找不到愿意配合的教师，这个项目的可行性要打问号。

交付：题型规则文档 + 测试题集 + Prompt v1 + 教师合作确认
2
模型评测与 prompt 迭代
Week 3

用同一题集对比 Claude Sonnet、GPT-4o、Gemini 的表现：准确率、步骤质量、越南语自然度、响应速度和成本。同时请教师对模型输出做首轮评分。根据评分结果选定主力模型并迭代 prompt。

交付：模型选择报告 + Prompt v2 + 教师首轮评分记录
3
MVP 开发
Week 4-5

用 Claude Code 初始化项目骨架，然后在 Cursor 里完成：移动端页面（首页 + 拍照/输入 + 题干确认 + 结果展示 + 反馈按钮）、API 路由（接入选定模型）、KaTeX 公式渲染、Supabase 日志记录。部署到 Vercel。

先保证"输入一道题 → 拿到答案 → 记录日志 → 能收反馈"这条链通了，再打磨 UI。

交付：可访问的 Web App + 移动端基本可用
4
质量闭环 + 教师审核
Week 6

按题型统计错误率，请教师做第二轮抽样评分。修正输出模板和 prompt 中的系统性问题。加入低置信题目的提示逻辑。

交付：Prompt v3 + 质量看板 v1 + 教师评分报告

种子用户验证

最后一句话

这个项目的目标不是做出一个"大而全的 AI 教育产品"。目标是在 8 周内回答一个问题：在越南高考数学这个具体场景里，一个编码了本地题型规则的产品，能不能比 Gauth 和 ChatGPT 做得明显更好？如果答案是能，继续往前走；如果答案是不能，趁早换方向，损失的只是 8 周时间。
